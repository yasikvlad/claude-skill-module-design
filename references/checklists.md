Валидационные чек-листы. Инструмент проверки: «у меня уже есть единица — это правильно сделанная единица?»

Не путать с определениями в `principles.md`: определение говорит, *что есть* единица, чек-лист — *как проверить*, что она корректна. **Если хотя бы на один вопрос ответ «нет» — единица собрана неверно, нужна перепроектировка.**

Чек-листы фрактальны: одна и та же структура на всех уровнях иерархии — описание / структура / ЦКП / SRP / взаимозаменяемость.

## Чек-лист модуля

1. Можно ли описать одним предложением, что модуль делает для своего заказчика?
2. Контракт описан явно и покрывает и успех, и ошибку?
3. У модуля есть ЦКП, нужный следующему модулю в цепочке (или ЦКП всей системы)?
4. Модуль меняется по одной причине? Если причин две — это два модуля.
5. Можно заменить модуль другим с тем же контрактом, не сломав соседей в цепочке?

### Пример прохождения

Модуль: `dna-parser`
1. ✓ «Принимает URL продукта и возвращает структурированный профиль»
2. ✓ Контракт: вход `URL`, выход `ProductProfile`, ошибка `ParseError` с типом и причиной
3. ✓ ЦКП — `ProductProfile`, нужен модулю `dna-analyzer`
4. ✓ Меняется при изменении формата источника
5. ✓ Любой парсер с тем же контрактом подойдёт

### Пример провала

Модуль: `dna-utils`
1. ✗ «Делает разные полезные штуки» — нет одного заказчика
2. ✗ Контракт размыт: разные функции с разными форматами
3. ✗ ЦКП непонятен
4. ✗ Меняется по любому поводу
5. ✗ Невозможно заменить — слишком много разнородных функций

**Решение:** разделить на модули по заказчикам. Каждый утилитарный набор — отдельный модуль с контрактом.

## Чек-лист этапа

1. Можно ли описать одним предложением, какую фазу трансформации этап реализует?
2. На входе этапа данные одной семантической природы, на выходе — другой?
3. У этапа есть собственный ЦКП, нужный следующему этапу или конечному ЦКП модуля?
4. Этап реализует ровно одну смену природы данных? Если две — это два этапа.
5. Можно заменить этап другим с тем же входом и выходом, не сломав соседние этапы внутри модуля?

### Пример прохождения

Этап: `harvesting` в модуле `dna-parser`
1. ✓ «Скачивает HTML по URL»
2. ✓ Вход: `URL`, выход: `HTML` — разные существительные, смена природы
3. ✓ ЦКП — `HTML`, нужен этапу `refinement`
4. ✓ Одна смена природы: URL → HTML
5. ✓ Любой загрузчик с тем же контрактом подойдёт (curl/requests/playwright)

### Пример провала

Этап: `process`
1. ✗ «Обрабатывает данные» — что значит «обрабатывает»?
2. ✗ Вход и выход одной природы — но внутри несколько трансформаций
3. ✗ ЦКП непонятен — есть промежуточные продукты
4. ✗ Реализует 2-3 смены природы (HTML → структура → нормализованная структура → валидированная)
5. ✗ Невозможно заменить — слишком много слитых функций

**Решение:** разделить на этапы по смене природы. Каждая смена существительного = граница нового этапа.

## Чек-лист операции

1. Можно ли описать одним предложением, какое действие операция выполняет?
2. Один вход и один выход?
3. Выполняет ровно одно действие?
4. Операция неделима — дальнейшее дробление сделает фрагменты неработоспособными?
5. Можно заменить операцию другой с тем же входом и выходом, не сломав соседние операции внутри этапа?

### Пример прохождения

Операция: `parse_html` в этапе `refinement`
1. ✓ «Парсит HTML в структурированный объект»
2. ✓ Вход: `HTML`, выход: `ParsedStructure`
3. ✓ Одно действие — парсинг
4. ✓ Дальнейшее дробление (`tokenize`, `build_tree`) — внутренняя реализация BeautifulSoup, не самостоятельные операции этапа
5. ✓ Любой HTML-парсер подойдёт

### Пример провала

Операция: `fetch_and_parse_and_validate`
1. ✗ «Загружает, парсит и валидирует» — три действия
2. ✗ Один вход, но три промежуточных выхода
3. ✗ Три действия в одной операции
4. ✗ Делится естественно на три операции (или три этапа, если меняется природа)
5. ✗ Замена требует понимания всех трёх внутренних шагов

**Решение:** разделить на три операции (или этапы, если каждое — смена природы данных). Проверь тестом существительного.

## Когда применять

- **Сразу после декомпозиции** — на каждом уровне иерархии прогоняй соответствующий чек-лист
- **Перед коммитом нового модуля** — финальная проверка перед фиксацией структуры
- **При рефакторинге** — когда подозреваешь, что граница проведена неверно; чек-лист подсветит, где именно
- **При обсуждении дизайна** — общий язык для команды: «модуль не проходит пункт 4 — давайте разделим»

Шаблон модуля. Каждый новый модуль в пайплайне стартует с этого чертежа.

## Принцип развёртывания

См. принцип «Иерархия и развёртывание» в `principles.md`. Краткая суть: иерархия Модуль → Этап → Операция действует на концептуальном уровне всегда, физическая структура модуля разворачивается ровно настолько, насколько того требует семантическая нагрузка задачи.

Ниже — две предельные точки развёртывания. Промежуточные формы допустимы. Условия выбора формы — в разделе «Когда разворачивать этап».

## Свёрнутая форма

Этапы существуют концептуально, но физически свёрнуты до файлов на уровне модуля. Папок этапов в структуре нет.

```
module_name/
├── __init__.py            ← публичный API: schemas + run
├── __main__.py            ← точка входа: python -m module_name
├── config.py              ← ModuleConfig (sub-config из PipelineConfig)
├── client.py              ← Gateway к внешнему сервису (транспорт)
├── storage.py             ← сохранение/загрузка данных
├── schemas.py             ← внутренний контракт модуля (Pydantic)
├── exceptions.py          ← контракт ошибок
├── validators.py          ← валидация входных параметров (если есть)
└── tests/
    ├── test_client.py
    ├── test_schemas.py
    └── test_validators.py
```

## Развёрнутая форма

Каждый этап развёрнут в собственную папку с оркестратором и (при необходимости) собственной триадой.

```
module_name/
├── __init__.py            ← публичный API: schemas + run
├── __main__.py            ← точка входа
├── config.py              ← ModuleConfig (root, содержит sub-configs этапов)
├── schemas.py             ← внутренний контракт модуля + межэтапные схемы
├── exceptions.py          ← контракт ошибок
│
├── stage_1/
│   ├── __init__.py
│   ├── orchestrator.py    ← оркестрация операций этапа
│   ├── <operation>.py     ← операция: один семантический шаг
│   ├── config.py          ← Stage1Config (sub-config из ModuleConfig)
│   ├── client.py          ← Gateway к внешнему сервису
│   └── storage.py         ← сохранение данных этапа
│
├── stage_2/
│   ├── __init__.py
│   ├── orchestrator.py
│   ├── <operation>.py
│   ├── config.py
│   ├── client.py
│   └── storage.py
│
└── tests/
    ├── test_schemas.py
    ├── stage_1/
    └── stage_2/
```

### Где живут операции

В свёрнутой форме операции могут жить:
- как функции в `client.py` — если операция = вызов внешнего сервиса (например, `generate_image` в `generation/client.py`);
- как методы класса-обёртки модуля (`analyzer.py`, `prompter.py`, `editor.py`) — если модуль stateful и инкапсулирует зависимости через конструктор;
- как отдельные файлы `<operation>.py` — если операция автономна и не зависит от client/storage.

В развёрнутой форме операции живут внутри `stage_*/` рядом с `orchestrator.py`, как отдельные файлы `<operation>.py`. Оркестратор координирует вызовы операций в этапе.

## Правила

### Триада (закон)

Состав и роли триады (Config, Client, Storage) описаны в `conventions.md`. Здесь — правила привязки триады к уровню внутри шаблона модуля.

**Где живёт триада:**
- если все этапы модуля используют общий внешний сервис — триада на уровне модуля;
- если этап работает с собственным внешним сервисом — триада на уровне этапа;
- если внешних сервисов нет — триады нет;
- комбинации допустимы: модульная триада для общего сервиса + этапная для специфичного. Уровни не исключают друг друга.

**Один владелец на каждый Config/Client/Storage.** Триада не дублируется и не размывается между уровнями: одна и та же настройка не может одновременно жить в Config модуля и Config этапа. Если возникает соблазн дублирования — граница проведена неверно.

**Триада не привязывается к операции.** Если операция нуждается в собственном Config/Client/Storage — это признак, что она по сути этап с одной операцией. Развернуть в этап.

**Инициализация:** Config → валидация (fail-fast) → инжекция в Client и Storage.

### Контракты

- **Внутренний контракт** (schemas.py) — структуры данных внутри модуля. Могут меняться свободно.
- **Публичный контракт** (contracts/ на уровне пайплайна) — вход и выход модуля. Изменение требует координации.
- Модуль импортирует публичные схемы из contracts/, не из соседних модулей.

### Именование

- Директория модуля: `dna-{name}/` (kebab-case)
- Python-пакет: `dna_{name}/` (snake_case)
- Config-класс: `{Name}Config`
- Этапы: по смыслу операции (`harvesting/`, `refinement/`, `analysis/`)

### Когда разворачивать этап

Этап существует концептуально всегда (см. принцип «Иерархия и развёртывание» в `principles.md`). Здесь — критерии его физического развёртывания в собственную папку.

Этап разворачивается, когда выполнено **хотя бы одно** из условий:
- этап имеет собственную координационную логику (orchestrator с условиями, циклами, ветвлениями) поверх вызовов своих операций;
- этап требует собственной триады (Config/Client/Storage), отличной от модульной;
- реализация этапа делает модуль трудночитаемым в одном файле;
- смысл этапа достаточно самостоятелен, чтобы иметь собственное имя в навигации (например, `harvesting/`, `refinement/`, `analysis/`).

Если ни одно условие не выполнено — этап остаётся свёрнутым до файлов на уровне модуля.

### Когда этап перерастает в модуль

Если этап становится нужен другим модулям напрямую — он перерос в модуль. Выноси его на уровень пайплайна.

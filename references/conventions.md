## Инфраструктурная триада

Config, Client, Storage — три служебных модуля для связи с тем, что снаружи: API, база данных, настройки среды.

1. Каждый отвечает за своё: настройки, связь, хранение.
2. Изменение одного не затрагивает остальные.

### Определения

**Config** — источник правды о настройках.
- Знает откуда брать параметры (env, файл, секреты).
- Не знает зачем они нужны — просто отдаёт значения.
- ЦКП: валидированные настройки, готовые к использованию.

**Client** — переводчик (Gateway) между модулем и внешним сервисом.
- Знает как общаться с API (формат запроса, авторизация, перевод терминов).
- Не знает бизнес-логику — только транспорт.
- Политики отказоустойчивости (retry, timeout, circuit breaker) — отдельный слой, который оборачивает Client, не смешиваясь с его логикой.
- ЦКП: ответ от внешнего сервиса в понятном формате.

**Storage** — хранитель состояния. Это роль, а не один файл.
- Для каждого типа хранилища (БД, кэш, файловая система) — свой модуль, реализующий общий контракт.
- Знает как сохранять и доставать данные.
- Не знает что эти данные значат — только где лежат и как достать.
- ЦКП: данные по запросу или подтверждение записи.

### Инициализация

Config загружается первым и валидируется при старте. Невалидный Config — приложение падает немедленно (fail-fast). Валидированный Config инжектируется в Client и Storage через конструктор.

Порядок: Config → валидация → Client, Storage.

### Принцип разделения

Каждый модуль триады меняется по своей причине:

- Config — новый параметр, другой источник. Пример: добавил новый API-ключ.
- Client — изменился внешний API. Пример: Claude обновил версию v2 → v3.
- Storage — другое хранилище. Пример: JSON → база данных Supabase.

### Правило

Одна причина для изменения — один модуль.

1. Поменялся API → трогаешь только Client.
2. Поменялось хранилище → трогаешь только Storage.
3. Добавил настройку → трогаешь только Config.

Если изменение требует правок в нескольких местах — граница проведена неверно. Исключение: сквозные аспекты (логирование, метрики, авторизация) по природе затрагивают несколько модулей. Это не ошибка границ — для них используются декораторы, middleware или отдельные сервисы.
